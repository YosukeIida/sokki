sokki — RecordingView

sokki は macOS 15+ 向けのネイティブ音声文字起こしアプリです。会議・インタビュー・講義を録音しながら、オンデバイス AI でリアルタイムに文字起こし・話者識別・翻訳を行います。このページはデザイン方針の共有とフィードバック収集のためのリビューページです。

バージョン: v2（配色・波形を再調整） 対象画面: RecordingView（録音中メイン画面） フェーズ: Phase 1 MVP プラットフォーム: macOS 15+ / Apple Silicon

アプリのコンセプト

sokki が解こうとしている課題と、それを解くためのアプローチ

🎙

録音しながら「読める」文字起こし

録音が終わったあとに文字起こしするのではなく、録音中にリアルタイムで文字が流れる。会議の途中でも「さっき何て言ったっけ」をスクロールで確認できる。

🔒

すべてオンデバイス処理

機密性の高い会話・インタビュー・社内ミーティングを扱う。音声データはクラウドに送らず、Apple Silicon の Neural Engine で完結。オンデバイスバッジが常時表示される。

👥

話者を識別して色分け

声紋から話者を自動識別し、色とラベル（Speaker A / B / 実名）で区別。多人数の議事録でも「誰が何を言ったか」が一目でわかる。

🌐

発話と同時に翻訳

日英・英日などのリアルタイム翻訳をオプションで表示。原文の直下に淡い訳文が流れる。外国語の会議でも「本来の言語」を残しながら理解できる。

🎵

Mic / System / Both の同軸波形

マイク入力（自分の声）とシステム音声（相手の声・PC の音）を別レイヤーで可視化。録音ソースの状態が一目で確認できる。

📄

Markdown エクスポート

録音終了後、話者・タイムスタンプ付きの Markdown で書き出せる。議事録・インタビューメモ・ノートに直接貼り付けられる形式。

デザイン思考 — 主な判断と根拠

各決定の「なぜ」。変更可能なものは 検討中 で表示

ふたつのモード、ひとつのアプリ： ライト（Manuscript）はドキュメント体験を重視した冷静な紙面、ダーク（Console）は暗い場所での長時間使用に耐える計器盤。配色は変わるが書体・構造・UX は同一。
決定事項 状態 採用した理由 検討した代替案 / 残る課題
書体を両モード統一（サンセリフ） 決定 Manuscript モードで明朝体を試したが、日本語小さいサイズでの可読性が低く、ラスタライズのばらつきが目立った。SF Pro / ヒラギノ角ゴで統一し、文書感は配色・余白・読み幅で表現する方針に。 Manuscript に限り serif を使う案は引き続きあり。フォントサイズが大きい一覧・詳細画面では再検討余地がある。
波形 = Voice Memos 風（対称・細い棒） 決定 録音時の入力レベル確認が主目的。装飾的な波形よりも実用的な「音が入っているか」の視認性を優先。3px 幅の棒＋隙間でノイズフロアを表現。 スペクトログラム、ルーメーター（VU）、Waveform ライン（単線）なども比較。現状 Mic 単体は1波形（対称）、Both は同軸2波形（mic=上 / sys=下）。
話者色を固定（声紋に紐づく） 決定 「Speaker A は青」というマッピングをセッション内で固定することで、発言者の視覚的追跡コストを下げる。色はライト・ダーク両方で WCAG AA を満たす中間色を選定。 色弱ユーザーへの配慮（形・パターン補助）は Phase 2 以降の課題。現状 3 話者まで色を定義（A=青、B=緑、C=紫）。4 話者以上の扱いは未決。
未登録話者 = Speaker A / B 表示 決定 声紋登録前でも話者が分かれていることを示す必要がある。実名が不明な場合のフォールバックとして汎用ラベルを使用。 「話者1・話者2」「?・??」「Unknown」なども比較。英語 "Speaker A" で統一することで、翻訳表示時も違和感が少ない。
ダーク配色: 青・赤を鎮静化 decided 初版の Mic 青（#5FA3FF）・録音赤（#FF4545）は暗背景で刺激が強すぎた。長時間使用での眼精疲労を考慮し彩度を低下（#6E96C9 / #D9534C）。 さらに下げる余地あり。アクセント色（ティール #35A597）との明度バランスは要確認。
翻訳はオリジナルの直下（orig → trans の順） 決定 原文を先に読んでから訳を補助として参照する使い方を想定。左ボーダーで「翻訳あり行」を視覚的にグループ化。 訳を先にする（trans → orig）案は多言語会議で役立つ可能性。切り替え可能なオプションにすべきか検討中。
オンデバイスバッジを常時表示 決定 プライバシーが sokki の差別化軸。「データが外に出ていない」ことを常に視認できることがユーザーの安心感に直結する。 バッジのサイズ・配置（ツールバー vs ステータスバー）は詰めが必要。
録音ボタン = 大きな丸形 検討中 「今録音中か」の状態視認性を最大化。VoiceOver ユーザーへの配慮として大ターゲットを確保。 macOS 標準 UI との乖離（ツールバーボタンが主流）が懸念。配置をフローティング vs ツールバーで比較したい。
3 ペイン構成（サイドバー + コンテンツ） 探索中 録音一覧・話者プロファイル・設定への常時アクセスを確保。macOS 標準 NavigationSplitView のパターンに沿っている。 録音中はサイドバーを自動で閉じる「集中モード」があると良いかもしれない。

オープンクエスチョン — デザイナーに意見を聞きたいこと

以下の点についてフィードバック・新しいアイデアを歓迎します

  • 1 録音ボタンの位置とサイズ：現在はコンテンツエリアの下部中央に大きく配置しています。macOS の標準ツールバーに収める案と、フローティングボタン案があります。「録音している」という状態感をどう表現すべきでしょうか？
  • 2 翻訳の表示方法：現在は原文の下に薄い字で訳を並べています。外国語会議では訳を主、原文を補助にしたいケースもあるかもしれません。どのような切り替え体験が自然でしょうか？
  • 3 色弱・アクセシビリティ：話者の識別を色に頼っています。色覚多様性ユーザーへの配慮として、色以外の補助（パターン・形・ラベル強調）をどう組み込むべきでしょうか？
  • 4 波形の情報量 vs 存在感：波形は「音が入っている」を確認するものですが、視覚的に目を引きすぎる可能性もあります。録音中のフォーカスは文字起こし側にあるはずです。波形をどの程度目立たせるべきでしょうか？
  • 5 話者が増えたときのスケーリング：現在 3 話者まで色を定義しています。4 人以上の会議ではどうすべきでしょうか（色を増やす・グループ化・ループ）？色の上限とラベル体験を提案してください。
  • 6 仮文字起こし（Hypothesis）の見せ方：現在リアルタイムに確定前テキストがカーソル付きで流れています。これが「入力中」なのか「AI が考え中」なのかが区別しにくいかもしれません。どのように表現すると良いでしょうか？
  • 7 次の画面：録音一覧・詳細 のデザイン方向：RecordingView の次は録音一覧（IndexView）と詳細（DetailView）です。同じ Manuscript / Console の 2 モードで横展開するとき、どんな差分・追加要素を入れると良いでしょうか？

フィードバック領域

以下の各エリアについて Issue / PR / コメントでフィードバックをどうぞ

🎨 カラーパレット

ライト（Manuscript）・ダーク（Console）それぞれのトーン、アクセント色、話者色のセット。WCAG コントラスト比や長時間使用時の疲労感についてのご意見を。

例: "ダークの録音赤がまだ強い" / "ライトの藍色はもう少し明るくて良い"

📐 レイアウトと情報密度

サイドバー幅・コントロールバーの高さ・トランスクリプトの行間・読み幅制限（Manuscript の max-width: 640px）などの空間設計。

例: "サイドバーは隠せる方が良い" / "コントロール部分が大きすぎる"

〰️ 波形デザイン

Voice Memos 風の棒グラフ波形について。視覚的な存在感・サイズ・Mic / System / Both の切り替え体験。代替案（ルーメーター・スペクトル・ライン波形など）のアイデアも歓迎。

例: "Both のとき上下が逆の方が直感的" / "波形は折りたたみ可能にすべき"

👤 話者 UI

Speaker A / B ラベル・話者ピン（色丸）・タイムスタンプ位置・話者リネームフロー（未実装）。色弱対応のアイデアも。

例: "名前が長いとき省略ルールが必要" / "話者クリックで詳細ポップオーバーを出すべき"

🌐 翻訳表示

原文→訳の表示順、訳の文字サイズ・色・左ボーダーによるグループ化。多言語ペアの切り替え UI（JA⇄EN 以外）。

例: "訳をトグルで全体 on/off したい" / "言語ペア切替はもっと目立つ場所に"

🔜 次の画面へのアイデア

録音一覧・詳細・エクスポート・話者プロファイル画面のデザイン方向について、新しいアイデアや参照したい先行事例などを。

例: "一覧は音声波形のサムネイル付きリスト" / "Reeder のようなスワイプアーカイブ"
フィードバックの送り方： GitHub Issues（ラベル design）に投稿するか、このファイルを直接編集して PR を送ってください。Figma ファイルを作成した場合はリンクを Issue に貼り付けていただけると助かります。

実装上の制約（デザイン参考情報）

デザイン提案の際に考慮していただきたい技術的な枠組み

🍎

SwiftUI + macOS 15+

UIKit は使わず SwiftUI のみ。macOS 15 以降の API のみ使用可能。iOS / iPadOS 対応は計画していない（macOS 専用）。

🧱

NavigationSplitView 構成

macOS 標準の 2 / 3 ペイン構成を前提としています。サイドバー・コンテンツエリア・インスペクタの 3 列が最大。

⚡

AVAudioEngine でリアルタイム処理

Phase 1 はマイク入力のみ AVAudioEngine で処理。Phase 2 以降は SCStream でシステム音声も取得。UI はバッファ単位の更新に対応する必要あり。

🤖

WhisperKit / SpeakerKit（オンデバイス）

文字起こしは WhisperKit、話者識別は SpeakerKit を使用。レイテンシが数百 ms〜数秒あるため、「仮文字起こし（Hypothesis）」→「確定文字起こし」の 2 段階表示が必要。
